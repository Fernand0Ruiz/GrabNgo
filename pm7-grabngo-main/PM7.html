Documentation: Architecture

GrabNGo continues to follow the Model-View-Controller architecture covered in lecture and homework. The interface elements are divided into two views (Menu and Cart), the buttons displayed on these views are treated as the controllers using add event listeners. The menu data is stored in the model, which consists of a menu object that has menu items nested in their own individual categories, for example, “burger”:{ Cheese_Burger:{ price: 2.00}.. }. The cart is also stored in the model with an array of added menu item objects that hold their name, price, and amount. Lastly, the user's inputted purchase information is stored in an object in the model. There are two pages index.html (main page), and confirmedOrderpage.html (which thanks the user for their purchase). Info.css holds all the styling for the pages, and the index.js holds the MVC/code for all the pages. The Architecture did not change from the previous iteration PM5. However, the page info.html (input info page/confirm order) was removed and made into a view for the checkout page, which follows the same structure as the cart view.

Documentation: Differences from Design

Due to a lack of time and experience, we had to remove navbar features and extra pages. In particular, we removed the search bar feature, because we could not figure out how to implement the feature to search through menu items. We also removed extraneous pages that required a lot of data to implement such as all the nutrition information for the menu items. In addition to the removal of components, we also added a few components to improve the design and functionality that were suggested from observations and the class presentation. One component that was added was a custom logo to make the interface feel more official and personable. Another component that was added was cart calculation rounding to get rid of the awkward trailing zeros. Many CSS changes were also implemented, such as column shadows, cart div spacing, and a CSS error fix on menu items being too small to use +/- functionality. Most importantly, the checkout page implementation was fully implemented and styled, versus the previous iteration where it was hardcoded on a separate page. Checkout implementation included displaying the items in one’s cart, the order total, an input box for special instructions, and a place order button that transitions the page to confirm that it was placed with an order number above.
Project Conclusion and Reflection.

1. The following changes should be made:

Search bar feature to cycle through menu items.

Credit/debit card payment feature, in addition to the already set pay pickup.

Additional data to the menu items, such as Nutrition and toppings.

Extra inputs, such as the ability to enter promo codes that automatically
apply to the cart.

Auto entering input, instead of update button to improve error prevention.


2. Maintaining page data would need to be added to the current prototype to make it able for production readiness testing. Without maintaining the state of the page the user will lose their data every time they transition to another page or refresh. Therefore, the page data must be maintained before testing to ensure that it is usable for the user.

3. The most fundamental key point learned from the design process was the importance of conducting observations and utilizing feedback. In particular, the design decision that should be avoided, are fueled by personal preference versus consideration for the user. For instance, in sketches, we preferred to increment and decrement the number of items from the item itself, however, we found that users preferred having multiple ways of controlling the quantity, so +/- buttons and quantity textboxes were added. A particularly helpful design decision stemmed from utilizing the design principles such as the idea of system visibility, which inspired us to make the cart visible to the user at all times. This feature received good feedback from our users and offered uniqueness compared to other online menu ordering systems. If someone were to work on a similar interface project I think they could learn from our experience that design decisions should be made based on the user, not the creators, and those design decisions should be focused on following the design principles of Shneiderman, Nielsen, and Norman.
